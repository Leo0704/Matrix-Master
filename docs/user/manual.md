## 1. 快速上手（15 分钟）

### 1.1 第一次启动

1. 双击主控图标启动。
2. 首次启动自动运行 setup wizard：
   - 检测 Python 后端
   - 自动建库（PostgreSQL + pgvector）
   - Headscale 部署引导（填 VPS 信息）
3. 进入主界面。

### 1.2 添加第一台手机

1. 准备一台 Android 手机（10+），装入 SIM 卡，仅用蜂窝数据。
2. 在手机端：
   - 安装 Tailscale app，登录团队账号
   - 安装 companion APK
   - 启动 companion，按提示授权无障碍服务
3. 在主控：
   - 左侧菜单"设备" → "添加设备"
   - 主控展示配对码 + 二维码
   - 手机端输入配对码（或扫码）
4. 等待"设备已添加"提示（约 10 秒）。

### 1.3 添加第一个账号

1. 主控 → "账号" → "添加账号"
2. 填写：
   - handle：小红书用户名
   - 选择 persona
   - 选择绑定设备
3. 在该手机上完成 XHS 登录（首次需人工）。
4. 主控显示账号已激活。

### 1.4 跑通第一个 Goal

1. 主控 → "目标" → "新建目标"
2. 自然语言输入："帮我每周发 3 篇美妆笔记"
3. 主控自动：
   - 检索知识库
   - 生成内容
   - 安排发布
4. 在"目标"页查看进度。

## 2. 主界面

### 2.1 顶部栏
- **状态指示**：主控 / 数据库 / Tailscale 三态
- **告警铃**：未处理告警数
- **用户菜单**：设置 / 退出

### 2.2 左侧导航
- **总览**：单屏 dashboard
- **设备**：设备列表 / 状态
- **账号**：账号列表 / 风险
- **目标**：Goal / Plan / Task 进度
- **内容**：笔记列表 / 日历

- **知识库**：persona / rule / topic 管理
- **告警**：告警列表 / 处理
- **设置**：配置 / 集成

## 3. 常用操作

### 3.1 添加 / 编辑 Persona

1. "知识库" → "Persona" → "新建"
2. 按 [kb-writing-guide §2.1](../testing/kb-writing-guide.md) 填写
3. 保存后 Agent 自动 rebuild 向量索引（约 1 分钟）
4. 用"知识库测试"检索 3 个查询验证

### 3.2 添加 / 编辑 Rule

1. "知识库" → "Rule" → "新建"
2. 选 category（forbidden / best_practice / limit_avoidance）
3. 写规则 + **判据**（关键，没有判据的规则 Agent 无法程序化执行）
4. 选 severity（1-5）

### 3.3 创建 Goal

#### 3.3.1 模板 Goal

- "本周美妆号净涨 500 粉"
- "每天发 1 篇数码测评"
- "本周互动率提升 20%"

#### 3.3.2 自定义 Goal

1. "目标" → "新建"
2. 自然语言描述（例："每月给母婴号写 30 篇笔记，每篇至少 500 字"）
3. Agent 解析并确认
4. 开始执行

### 3.4 监控设备

1. "设备"页面
2. 状态颜色：
   - 绿：在线 + 健康
   - 黄：tailscale_degraded
   - 灰：offline
   - 红：error
3. 点击设备 → 详情：心跳历史、电池、流量、最近任务

### 3.5 接管 / 暂停

- 异常任务 → 详情页 → "接管"（人工在手机端操作）
- 暂停账号 → "账号" → 详情 → "暂停"

### 3.6 应急停止

- 全局暂停：设置 → "紧急停止所有 Goal"（Agent 进入 IDLE，不再触发新 run）

## 4. 自然语言指令参考

主控支持自然语言交互。常用指令：

| 意图 | 示例 |
|---|---|
| 创建 Goal | "帮我这周给美妆号发 5 篇笔记" |
| 查询状态 | "美妆号现在发了几篇了？" |
| 修改策略 | "美妆号改用更活泼的语气" |
| 暂停 | "暂停所有美妆号" |
| 重启 | "重新启动美妆号的 Goal" |
| 数据 | "美妆号最近 7 天的阅读量趋势" |
| 接管 | "接管美妆号 A" |
| 紧急 | "立刻停止所有 Goal" |

Agent 解析后会反馈确认，再执行。

## 5. 告警处理

### 5.1 告警类型

| 告警 | 含义 | 行动 |
|---|---|---|
| `DEVICE_OFFLINE` | 设备 5+ 分钟无心跳 | 查手机 / 查 Tailscale |
| `RISK_BLOCKED` | 账号触发风控 | 暂停账号 + 人工检查 |
| `SELECTOR_NOT_FOUND` | 选择器失效 | 人工介入或等修复 |
| `TASK_TIMEOUT` | 任务超时 | 查手机 / 查网络 |

### 5.2 处理流程

1. 收告警（系统通知 / 邮件）
2. 点击告警 → 详情（含上下文、trace ID）
3. 选择：
   - 重试
   - 跳过
   - 接管
   - 标记已处理

## 9. 进阶

### 9.1 知识库优化

定期审查 persona / rule：
- 哪些 rule 频繁被触发？（可能太严）
- 哪些 rule 从不触发？（可能没用）
- 哪些 persona 表现好？（借鉴）

详见 [kb-writing-guide §8](../testing/kb-writing-guide.md)。

### 9.2 性能调优

- 调设备并发度（设置 → 调度）
- 调活跃窗（避开平台高峰）
- 调 LLM 模型（成本 vs 质量）

### 9.3 数据导出

- 笔记 CSV
- 账号指标 CSV
- 审计日志（合规需要时）
